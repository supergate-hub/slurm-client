## Problem Statement

slurm-client v2.0.0은 단일 Slurm 클러스터용 Go SDK + MCP Server를 제공한다. 하지만 실제 HPC 환경에서는 여러 클러스터를 운영하는 것이 일반적이며, AI 에이전트가 "어느 클러스터에서?" 라는 질문을 자연스럽게 처리할 수 없다. 또한 MCP 툴은 기본 CRUD에 머물러 있어 GPU 모니터링, 작업 분석 등 고급 기능이 부족하다.

## What Makes This Cool

- **openstacksdk의 Cloud 추상화를 Slurm에 적용**: `client.Cluster("gpu-cluster").Slurm.Jobs().List(ctx)` 같은 자연스러운 멀티 클러스터 접근
- **AI가 HPC를 자연어로 제어하는 최초의 프로덕션급 MCP Server**: "GPU 클러스터에서 유휴 노드 확인해줘" → 실제 실행
- **Slurm 생태계에서 유일한 ergonomic Go SDK**: OpenAPI 자동생성이 아닌, openstacksdk 스타일의 사용자 경험

## Constraints

- 기존 v2.0.0 단일 클러스터 API와의 하위 호환성 유지
- Go 모듈 시맨틱 버저닝 준수. 현재 go.mod에 major version suffix 없음 (v0/v1 체계). Manager 추가가 breaking change가 아니면 v2.x로 유지 가능. breaking change 시 `/v3` suffix 도입 필요 (모듈 경로 마이그레이션 포함)
- MCP 프로토콜 표준 준수 (mcp-go 라이브러리 사용)
- Slurm REST API 버전 v0039~v0044 모두 지원 (코드: `api/v0039/` ~ `api/v0044/`)

## Premises

1. Slurm 생태계에 어고노믹한 Go SDK가 필요하다 (OpenAPI 자동생성 클라이언트는 사용 경험이 나쁘다)
2. AI 에이전트가 HPC 클러스터를 직접 제어하는 수요가 있다 (MCP Server)
3. 멀티 클러스터 지원과 MCP 고도화가 다음 단계로 적합하다
4. 배포 채널: Go 모듈(go get) + MCP 바이너리(GitHub Releases)

## Approaches Considered

### Approach A: Incremental Enhancement
기존 단일 클러스터 Client에 점진적으로 멀티 클러스터 지원 추가. 최소 diff.
- Effort: S | Risk: Low
- Pros: 빠른 배포, 기존 코드 최대 활용
- Cons: 12개월 후 리팩토링 필요, 멀티 클러스터가 애드혹으로 느껴짐

### Approach B: Multi-Cluster First Architecture (CHOSEN)
MultiClient 추상화 도입. openstacksdk의 Connection/Cloud 패턴처럼 여러 클러스터를 하나의 인터페이스로 관리. MCP도 멀티 클러스터 네이티브.
- Effort: M | Risk: Med
- Pros: 장기적 최적 아키텍처, 자연스러운 멀티 클러스터 UX
- Cons: 더 큰 변경, API 호환성 설계 필요

### Approach C: MCP-Centric
SDK 안정화 후 MCP Server를 핵심 제품으로 집중.
- Effort: S | Risk: Low
- Pros: 차별점(MCP)에 집중, 넓은 사용자층
- Cons: SDK 정체 가능

## Recommended Approach

**Approach B: Multi-Cluster First Architecture**

### SDK Layer

```
// 현재 (v2.0.0) — 단일 클러스터
client, _ := slurmclient.NewClient(ctx, slurmclient.AuthOpts{
    Endpoint: "http://cluster1:6820",
    AuthToken: "token",
})
jobs, _ := client.Slurm.Jobs().List(ctx)

// 제안 (v2.x additive) — 멀티 클러스터
manager, _ := slurmclient.NewManager(ctx, slurmclient.ManagerOpts{
    Clusters: map[string]slurmclient.AuthOpts{
        "gpu":  {Endpoint: "http://gpu-cluster:6820", AuthToken: "token1"},
        "cpu":  {Endpoint: "http://cpu-cluster:6820", AuthToken: "token2"},
    },
})

// 특정 클러스터 접근 — *Client 직접 반환 (기존 API 동일)
client := manager.Cluster("gpu")  // returns *Client
jobs, _ := client.Slurm.Jobs().List(ctx)

// 전체 클러스터 병렬 조회 — Go generics 제약으로 fluent chaining 불가
// 명시적 편의 함수 패턴 사용:
type ClusterResult[T any] struct {
    Cluster string
    Data    T
    Err     error  // 개별 클러스터 실패 시 partial result
}
allJobs, _ := manager.AllJobs(ctx)    // []ClusterResult[[]Job]
allNodes, _ := manager.AllNodes(ctx)  // []ClusterResult[[]Node]
// errgroup 병렬 실행, 일부 실패해도 성공한 결과는 반환

// 범용 패턴 (커스텀 쿼리):
results := manager.ForEach(ctx, func(name string, c *Client) (any, error) {
    return c.Slurm.Jobs().List(ctx)
})

// 각 Client는 독립적으로 API 버전 협상 (기존 동작 유지)
// Manager는 클러스터 간 버전 정규화를 시도하지 않음

// 하위 호환: 단일 클러스터는 기존 API 그대로 동작
client, _ := slurmclient.NewClient(ctx, slurmclient.AuthOpts{...})  // 여전히 동작
```

### MCP Layer

```
기존 24개 툴 유지 + 멀티 클러스터 확장:

새 기본 툴:
- slurm_list_clusters          // 연결된 클러스터 목록
- slurm_cluster_status         // 클러스터별 상태 요약
- slurm_cross_cluster_jobs     // 전 클러스터 작업 조회

고도화 1단계 (slurmrestd 데이터만으로 구현 가능):
- slurm_node_health_summary    // 노드 상태 요약 (healthy/drain/down) — nodes API로 구현
- slurm_resource_availability  // 클러스터/파티션별 가용 리소스 현황 — 추천 판단은 AI 에이전트에 위임

고도화 2단계 (GRES 데이터 + 휴리스틱 필요):
- slurm_gpu_allocation         // GPU GRES 할당 현황 (allocated/total) — slurmrestd GRES 정보 기반
- slurm_queue_depth            // 큐 대기 작업 수/우선순위 분포 — squeue 정보 기반

NOTE: 실시간 GPU utilization(nvidia-smi 수준)은 slurmrestd 범위 밖.
      필요시 Prometheus/DCGM 연동을 별도 단계로 계획.
```

## Design Decisions (resolved)

1. **버전 전략**: Manager는 기존 Client와 공존 가능하므로 additive change. v2.x로 유지하되, Manager 인터페이스가 안정되면 v3.0.0으로 promote. go.mod에 major suffix 추가는 v3 시점에 수행.
2. **멀티 클러스터 에러 핸들링**: `ClusterResult[T]` 래퍼로 partial result 반환. 개별 클러스터 실패는 해당 entry의 Err 필드에 기록, 성공한 클러스터 결과는 정상 반환.
3. **MCP 멀티 클러스터 설정**: config 파일 기반 (`--config clusters.yaml`). 단일 클러스터는 기존 환경변수(`SLURM_ENDPOINT`, `SLURM_TOKEN`) 하위 호환.

```yaml
# clusters.yaml 예시
clusters:
  gpu:
    endpoint: http://gpu-cluster:6820
    token: ${GPU_SLURM_TOKEN}  # 환경변수 참조 가능
    version: v0.0.40            # optional, default: auto-detect
  cpu:
    endpoint: http://cpu-cluster:6820
    token: ${CPU_SLURM_TOKEN}
```

## Success Criteria

- 멀티 클러스터 Manager가 2개 이상 클러스터에서 동시 조회 가능
- 기존 단일 클러스터 Client API 100% 하위 호환
- MCP Server에서 멀티 클러스터 툴 최소 3개 동작
- MCP 고도화 분석 툴 최소 2개 동작
- 통합 테스트로 실제 slurmrestd 검증 (기존 integration_test.go 확장)

## Distribution Plan

- **SDK**: `go get github.com/supergate-hub/slurm-client` (v2.x 유지, Manager는 additive). v3.0.0 시 `go get github.com/supergate-hub/slurm-client/v3` + go.mod 경로 마이그레이션 필요
- **MCP 바이너리**: GitHub Releases로 배포, goreleaser 활용
- **CI/CD**: GitHub Actions — 테스트 → 빌드 → 릴리즈 자동화

- config YAML의 `${ENV_VAR}` 참조 파싱은 별도 구현 필요 (viper 등 라이브러리 또는 직접 구현)
